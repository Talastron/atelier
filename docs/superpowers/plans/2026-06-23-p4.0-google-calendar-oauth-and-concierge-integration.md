## Decisions already made (don't re-debate)

1. **Provider:** Google Calendar only for v1. Microsoft Outlook + Apple deferred — they need separate OAuth/CalDAV implementations.
2. **Backend:** Firebase Cloud Functions. Refresh tokens MUST be server-side. The Firebase Blaze plan is active; Functions are available without additional setup.
3. **Privacy scope:** `https://www.googleapis.com/auth/calendar.events.readonly` — minimum that lets us list events. User-facing copy must say "read-only, we never edit or delete."
4. **First-launch UX:** Optional, not required. The app works fine without calendar; connection lives in Profile.
5. **No event-text leaves the device unless the user is actively in a Concierge/Daily Brief flow.** Events are fetched on demand for AI prompts, not continuously streamed.
6. **Caching:** 15-minute server-side cache per user per day. Calendar API has quotas; constant refetching is wasteful.
   - **STATUS (impl):** NOT YET implemented server-side — `getCalendarEvents` hits Google on every call. Mitigated for now by a per-session, per-date client cache in `WearCalendar` (`eventCacheRef`). Fine at single-user scale (Google quota is 1M queries/day). Server-side cache is a deferred optimisation — revisit before onboarding multiple users.

### Task 1.2: OAuth start endpoint

Write `calendarOAuthStart` — an HTTPS callable function that:
1. Verifies the caller is authenticated (request.auth.uid present)
2. Generates a state token (random string), saves it to `/users/{uid}/integrations/_oauth_state` with a 10-minute TTL
3. Constructs the Google OAuth URL with:
   - `client_id` from secrets
   - `redirect_uri` from secrets
   - `scope` = `https://www.googleapis.com/auth/calendar.events.readonly`
   - `state` = the token
   - `access_type=offline` (so we get a refresh token)
   - `prompt=consent` (force consent screen even on re-auth, to ensure refresh token)
4. Returns `{ authUrl }` — the client opens it in a popup or redirects to it

Inline code in `functions/index.js`:

```javascript
const { onCall, HttpsError } = require('firebase-functions/v2/https');
const { onRequest } = require('firebase-functions/v2/https');
const { defineSecret } = require('firebase-functions/params');
const admin = require('firebase-admin');
const { google } = require('googleapis');
const crypto = require('crypto');

admin.initializeApp();

const GOOGLE_OAUTH_CLIENT_ID = defineSecret('GOOGLE_OAUTH_CLIENT_ID');
const GOOGLE_OAUTH_CLIENT_SECRET = defineSecret('GOOGLE_OAUTH_CLIENT_SECRET');
const GOOGLE_OAUTH_REDIRECT_URI = defineSecret('GOOGLE_OAUTH_REDIRECT_URI');

exports.calendarOAuthStart = onCall(
  { secrets: [GOOGLE_OAUTH_CLIENT_ID, GOOGLE_OAUTH_REDIRECT_URI] },
  async (request) => {
    if (!request.auth?.uid) throw new HttpsError('unauthenticated', 'Sign in required.');
    const uid = request.auth.uid;
    const state = crypto.randomBytes(32).toString('hex');
    await admin.firestore().doc(`users/${uid}/integrations/_oauth_state`).set({
      state,
      createdAt: Date.now(),
    });
    const oauth2 = new google.auth.OAuth2(
      GOOGLE_OAUTH_CLIENT_ID.value(),
      undefined, // secret not needed for URL construction
      GOOGLE_OAUTH_REDIRECT_URI.value(),
    );
    const authUrl = oauth2.generateAuthUrl({
      access_type: 'offline',
      prompt: 'consent',
      scope: ['https://www.googleapis.com/auth/calendar.events.readonly'],
      state: `${uid}.${state}`, // include uid so callback can route
    });
    return { authUrl };
  }
);
```

### Task 1.3: OAuth callback endpoint

Write `calendarOAuthCallback` — an HTTPS request handler (NOT callable) that Google redirects to with `code` and `state`:

```javascript
exports.calendarOAuthCallback = onRequest(
  { secrets: [GOOGLE_OAUTH_CLIENT_ID, GOOGLE_OAUTH_CLIENT_SECRET, GOOGLE_OAUTH_REDIRECT_URI] },
  async (req, res) => {
    const code = req.query.code;
    const stateParam = req.query.state;
    if (!code || !stateParam) {
      return res.status(400).send('Missing code or state');
    }
    const [uid, state] = String(stateParam).split('.');
    if (!uid || !state) return res.status(400).send('Invalid state');

    // Verify state matches
    const stateDoc = await admin.firestore().doc(`users/${uid}/integrations/_oauth_state`).get();
    if (!stateDoc.exists || stateDoc.data().state !== state) {
      return res.status(403).send('State mismatch');
    }
    // Optional: check createdAt < 10min ago

    // Exchange code for tokens
    const oauth2 = new google.auth.OAuth2(
      GOOGLE_OAUTH_CLIENT_ID.value(),
      GOOGLE_OAUTH_CLIENT_SECRET.value(),
      GOOGLE_OAUTH_REDIRECT_URI.value(),
    );
    const { tokens } = await oauth2.getToken(code);

    // Split storage so the SPA can never read tokens.
    //
    //   users/{uid}/integrations/google_calendar      — public-readable metadata
    //                                                   (scope, connectedAt)
    //   integrationSecrets/{uid}/providers/google_calendar — server-only
    //                                                   (refresh/access/expiry)
    //
    // Tokens live OUTSIDE /users/{uid} because Firestore rules are ADDITIVE:
    // the recursive /users/{uid}/{document=**} allow can't be narrowed by a
    // more-specific deny. A path no client-allow rule matches (default-deny +
    // explicit deny on /integrationSecrets) is the only way to truly seal it.
    // Functions reach it via the Admin SDK, which bypasses rules.
    await admin.firestore().doc(`integrationSecrets/${uid}/providers/google_calendar`).set({
      refreshToken: tokens.refresh_token,
      accessToken: tokens.access_token,
      expiresAt: tokens.expiry_date,
    });
    await admin.firestore().doc(`users/${uid}/integrations/google_calendar`).set({
      scope: tokens.scope,
      connectedAt: Date.now(),
    });

    // Clean up state token
    await admin.firestore().doc(`users/${uid}/integrations/_oauth_state`).delete();

    // Redirect back to the app with a success flag
    res.redirect('https://edit.myatelier.style/?calendarConnected=1');
  }
);
```

NOTE: in dev, the redirect URI is the Functions emulator, and the post-callback redirect should go to `http://localhost:5173`. Handle both by reading `req.headers.origin` or checking an env var.

**Region:** All three Functions are deployed to `europe-west2`. Add `region: 'europe-west2'` to each `onCall`/`onRequest` options object (alongside `secrets`).

### Task 1.4: Event fetch endpoint

```javascript
exports.getCalendarEvents = onCall(
  { secrets: [GOOGLE_OAUTH_CLIENT_ID, GOOGLE_OAUTH_CLIENT_SECRET, GOOGLE_OAUTH_REDIRECT_URI] },
  async (request) => {
    if (!request.auth?.uid) throw new HttpsError('unauthenticated', 'Sign in required.');
    const uid = request.auth.uid;
    const { startISO, endISO } = request.data; // YYYY-MM-DD strings

    const integrationDoc = await admin.firestore()
      .doc(`users/${uid}/integrations/google_calendar`).get();
    if (!integrationDoc.exists) {
      throw new HttpsError('failed-precondition', 'Calendar not connected.');
    }
    const integration = integrationDoc.data();

    // Set up OAuth client with stored tokens
    const oauth2 = new google.auth.OAuth2(
      GOOGLE_OAUTH_CLIENT_ID.value(),
      GOOGLE_OAUTH_CLIENT_SECRET.value(),
      GOOGLE_OAUTH_REDIRECT_URI.value(),
    );
    oauth2.setCredentials({
      refresh_token: integration.refreshToken,
      access_token: integration.accessToken,
      expiry_date: integration.expiresAt,
    });

    // googleapis auto-refreshes if access token expired
    const calendar = google.calendar({ version: 'v3', auth: oauth2 });
    const result = await calendar.events.list({
      calendarId: 'primary',
      timeMin: new Date(startISO + 'T00:00:00').toISOString(),
      timeMax: new Date(endISO + 'T23:59:59').toISOString(),
      singleEvents: true,
      orderBy: 'startTime',
      maxResults: 50,
    });

    // If the access token was refreshed, persist the new one
    const updatedCreds = oauth2.credentials;
    if (updatedCreds.access_token !== integration.accessToken) {
      await integrationDoc.ref.update({
        accessToken: updatedCreds.access_token,
        expiresAt: updatedCreds.expiry_date,
      });
    }

    // Project events to a minimal shape — title, time, location ONLY.
    // Strip descriptions/attendees/anything sensitive.
    return {
      events: (result.data.items || []).map((e) => ({
        id: e.id,
        title: e.summary || '(untitled)',
        startISO: e.start?.dateTime || e.start?.date,
        endISO: e.end?.dateTime || e.end?.date,
        allDay: !!e.start?.date && !e.start?.dateTime,
        location: e.location || null,
      })),
    };
  }
);
```

From Profile → Disconnect. Verify Firestore document is deleted. Subsequent event fetches throw `failed-precondition` and the UI shows the "Connect Google Calendar" state.

### Task 4.3: Revoked access

User revokes access from https://myaccount.google.com/permissions. Next event fetch will throw a 401 from Google. The function should catch this and either:
- Delete the integration doc and return `{ events: [], reason: 'revoked' }`
- Or surface the error to the client which then prompts re-connection

### Task 4.4: No events

Empty result from Google = empty array. UI hides the "On your calendar" block. AI prompts get an empty events block (the `?.length > 0` guard skips it).

### Task 4.5: Multi-calendar

For v1: primary calendar only. Multi-calendar selection (work + personal + family) is a follow-up. Doc says "primary calendar" in the Profile UI.

### Task 4.6: Timezone

Calendar API returns events in the calendar's timezone. Render times in the user's locale via `Date.prototype.toLocaleTimeString`. Trust the API's `dateTime` strings — they're ISO with offset.

## Risks

1. **OAuth verification**: To accept non-test-user accounts, Google requires app verification (privacy policy, terms, scope justification, security assessment). Until verified, only listed test users can connect. Plan: ship for personal use first, submit verification when ready for other founders.

2. **Cloud Functions cold start**: First-call latency can be 2-4s. Acceptable for connection (one-time) but might be noticeable on event fetch. Mitigate with `minInstances: 1` on `getCalendarEvents` (small ongoing cost).

3. **Refresh token loss**: If `refresh_token` is null in the OAuth response (happens when user has previously authorised), the integration doc is unusable. The `prompt=consent` flag in `calendarOAuthStart` mitigates by forcing a fresh consent which always returns a refresh token.

4. **Quota**: Calendar API has a per-project daily quota (1 million queries by default). Won't be a problem at single-user scale.
